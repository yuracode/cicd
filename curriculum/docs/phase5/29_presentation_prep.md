# コマ29｜発表準備（README・スライド）

| 項目 | 内容 |
|------|------|
| フェーズ | Phase 5 |
| 所要時間 | 90分 |
| 前提コマ | コマ28 個人制作④：CI/CD適用 |
| 次コマ | コマ30 最終発表・振り返り |

##  目標

- 5分の発表用スライドを作成できる
- 30秒／1分／3分 の3レンジで自分のプロジェクトを説明できる
- 発表リハーサルで時間を確認し、本番で焦らない準備を整える

##  導入（15分）

### 次回が発表

発表時間：**1人5分**（プレゼン3〜4分＋質疑応答1〜2分）

### 発表で評価されるポイント

1. **何を作ったかが伝わるか**
2. **CI/CDパイプラインを説明できるか**（今回の授業の中心テーマ）
3. **デモで動いているものが見られるか**
4. **自分の言葉で話せているか**

### よくある失敗

-  スライドを読むだけ（原稿棒読み）
-  デモに時間をかけすぎる（本体機能じゃない部分の説明が長い）
-  時間オーバー
-  CI/CDについて一切触れない

##  本題（65分）

### 1. スライド構成を決める（10分）

**5分発表の黄金構成：**

| 時間 | 内容 |
|------|------|
| 0:00〜0:30 | タイトル・何を作ったか一言 |
| 0:30〜1:30 | デモ（触って見せる） |
| 1:30〜2:30 | 技術スタックとアーキテクチャ |
| 2:30〜3:30 | **CI/CDパイプライン**（ここが主役） |
| 3:30〜4:00 | 苦労した点・工夫した点 |
| 4:00〜5:00 | 質疑応答 |

### 2. スライドツールを選ぶ（5分）

どれでもOK：

- **Google Slides**（共有しやすい）
- **PowerPoint**
- **Keynote**（Mac）
- **Marp**（Markdownでスライド、エンジニア向き）

> **Marp を使う場合：** VS Code拡張「Marp for VS Code」を入れると `.md` からスライドPDFが生成できる。

### 3. スライドを作る（25分）

**スライド1：タイトル**

```
 Pomodoro App
技術研究 発表

学籍番号 名前
```

**スライド2：どんなアプリか**

- 1文で要約
- ターゲットユーザー
- スクリーンショット

**スライド3：デモ**
（スライドは小さく、実画面をメインに）

- 開いているだけの軽いスライドにし、ブラウザに切り替えて触る

**スライド4：技術スタック**

```
Frontend:  React 19 / Vite / JavaScript
Test:      Vitest / React Testing Library
Quality:   ESLint / Prettier
CI/CD:     GitHub Actions
Hosting:   Vercel (Production) / GitHub Pages
```

**スライド5：CI/CDパイプライン（ここが重要）**

パイプライン図を必ず入れる：

```
push / PR
  ↓
[Lint][Test][Build]  ← 並列実行
  ↓ 全て緑のみ通過
[Deploy]
  ├─ Vercel（PRプレビュー + 本番）
  └─ GitHub Pages（main のみ）
```

**スライド6：工夫／苦労ポイント**

- 「useEffectの依存配列に悩んだ」
- 「Fake timer の設定で半日詰まった」
- 「CIが CI独自の環境でFAIL」
- など、**正直に詰まったポイントを話す方が好印象**

**スライド7：まとめ**

- 公開URL（QRコードもあると親切）
- GitHub URL
- ありがとうございました

### 4. 発表原稿を書く（10分）

箇条書きベースで、**丸暗記ではなくキーワード** を書く。

```markdown
# 発表原稿メモ

## タイトル（10秒）
こんにちは、〇〇です。Pomodoro アプリを作りました。

## どんなアプリ（30秒）
25分集中＋5分休憩を自動で繰り返すタイマーアプリです。
私自身が試験勉強で使っていて、ブラウザで1クリックで使えるものが欲しくて作りました。

## デモ（1分）
（画面を見せながら）
- 開始ボタンでカウントダウン
- 0になると休憩モードへ
- リセットで最初に戻る

## 技術スタック（30秒）
React 19 と Vite で作っています。
テストは Vitest と React Testing Library を使いました。

## CI/CD（1分）  ← 長めに
GitHub Actions で Lint・テスト・ビルドを並列で実行しています。
全て緑の時だけ main にマージでき、マージされると自動で Vercel にデプロイされます。
PR時には別URLで試せるプレビュー環境が自動で立ち上がります。

## 工夫（30秒）
タイマーのテストで fake timer の設定に苦労しましたが、
Phase 2 で学んだ useFakeTimers で解決できました。

## まとめ（20秒）
公開URLとGitHubはこちらです。ご質問お願いします。
```

### 5. リハーサル：1人で時間を計る（10分）

タイマーをセットして、話してみる。

- **時間オーバーを警戒**：早めに終わる方が安全
- **詰まる箇所を特定**：何度も引っかかる所はキーワードを足す
- **話すスピード**：1分間に **300〜400文字程度** が聴きやすい

**スクリーンとPCの切り替え動作** も練習：

- ブラウザを事前に開いておく（大きい文字・ブックマーク準備）
- `Alt + Tab` で素早く切り替える

### 6. 隣の学生と相互デモ（5分）

2人ペアで発表し合う：

- 聞き手：「分かりやすかった点」「分からなかった点」を1つずつ言う
- 話し手：メモを取って改善点を修正

### 7. 最終チェックリスト（5分）

発表前に必ず確認：

- [ ] スライドが最新
- [ ] 公開URL（Vercel / GitHub Pages）が動く
- [ ] GitHub リポジトリが閲覧可能（public）
- [ ] CIの Actions タブに緑の履歴
- [ ] READMEの見た目（スクリーンショット反映済み）
- [ ] スライドは時間通り終わる
- [ ] （可能なら）QRコードで公開URL／GitHubを見せられる

QRコード生成：

- https://www.qrcode-monkey.com/ など無料ツール
- スライドに貼っておくと質疑応答で便利

##  まとめ（10分）

### 今日できたこと

- 5分発表のスライドが完成
- 発表原稿のキーワードメモができた
- 1人リハーサル＆相互リハーサルで時間感覚がついた

### 発表の心構え

- **「技術の凄さ」より「自分の言葉で語れるか」** が評価される
- **詰まった経験は語る価値あり**。完璧な発表より誠実な発表
- **時間内に終わる** のが最優先。2分の話を5分でしても悪印象

### 次コマ予告

次回が最終回、いよいよ発表。全員の発表を見て、互いに学び、振り返る。

##  課題

### 基礎課題（必須）

1. **家族・友人に対して発表のリハーサル** をする（相手がエンジニアでなくても「何を作ったか」は伝わるように）
2. スライドをPDF化して、発表当日は **オフライン再生** でも見られるようにしておく
3. 質疑応答の **想定問答** を3つ準備（「どうしてこのアプリを？」「一番難しかったのは？」「次作るなら何？」など）

### 応用課題（推奨）

4. **発表原稿に「詰まった経験」を1つ必ず入れる**：何に何分（または何時間）詰まり、どう解決したか。技術的な失敗談は発表の説得力を大きく上げる。完璧な物語より、**具体的な苦労話** のほうが人の記憶に残る

5. **技術スタックのロゴ** をスライドに視覚的に配置する：React/Vite/Vitest/GitHub Actions/Vercel 等の公式ロゴをスライドに並べる。テキストだけより **直感的に理解** される

6. **「3分縮小版」の原稿** も準備する。本番5分だが、機材トラブルや前者延長で時間が削られた時用：
   - 5分版：前置き＋デモ＋詳細＋苦労話＋今後
   - 3分版：デモ＋苦労話＋今後

どちらのパターンでも話せるようにしておく。

> **狙い：** 本番は何があるかわからない。**複数パターンの準備** が実務のプレゼンでは標準。

### チャレンジ課題（挑戦）

7. **想定外の質問への答え方パターン** を1つ準備する：
   - 「分かりません、すみません」は **失礼ではない**。むしろ誠実
   - 「良いご指摘です、持ち帰って調べます」
   - 「今回は〜を優先したため、〜は範囲外としました」

→ エンジニアとして「分からないことを分からないと言える」のは重要なスキル。

8. **同期の発表を聞くチェックリスト** を自分用に作る：
   - 何を作ったか30秒で説明できるか
   - デモで何を見せたか
   - CI/CDの組み方で参考にしたい部分は？
   - 自分なら追加したい機能は？

当日、他人の発表を能動的に聞くためのテンプレート。

9. **本番環境に最終チェック**：発表直前にURLを叩いて動作確認。localStorage が意図通り、バッジが緑、READMEに誤字なし、を再確認
