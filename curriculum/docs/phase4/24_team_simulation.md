##  目標

- 3人チームで1つのリポジトリに相互PRを出しあい、レビューしてマージできる
- マージ競合（コンフリクト）を自分で解決できる
- レビューコメントの書き方・受け方のマナーを体験する

##  導入

### Phase 4 までの到達

個人としては CI/CD パイプラインを組める状態になった。今日は **チームで動かす** ことを体験する。

### 今日のシミュレーション設定

- 3人チームを作る（人数が違ったら2人でもOK）
- 1つの **共有リポジトリ** を用意（誰か1人の既存 `todo-app` を使う）
- 全員Collaboratorとして招待
- 各自「新機能ブランチ」を切って実装 → PR → 相互レビュー → マージ

### シミュレーションで体験すること

1. 共同作業でのブランチ戦略
2. PRコメントのやりとり
3. コンフリクト解決
4. ブランチ保護の「承認レビュー必須」ルールの体感

##  本題

### 1. チーム作り＆リポジトリ招待

チーム代表の人が：

```bash
cd ~/workspace/todo-app
gh repo view --web
```

**Settings → Collaborators → Add people** で他の2人のGitHubアカウントを追加。

招待されたメンバーはメール通知を確認して Accept。

招待されたメンバーは **手元にクローン**：

```bash
cd ~/workspace
git clone https://github.com/代表のユーザー名/todo-app.git
cd todo-app
npm ci
npm run dev   # 動くことを確認
```

### 2. ブランチ戦略の合意

ホワイトボード（またはチームチャット）で決めておく：

- ブランチ命名規則：`feature/名前-機能名`（例：`feature/tanaka-filter`）
- mainは保護（コマ17で設定済み）
- PRマージ条件：**承認1以上＋CI全パス**
- コミットメッセージ：Conventional Commits

### 3. 各自の機能分担

例（3人チーム想定）：

| 担当 | 機能 |
|------|------|
| Aさん | **フィルタ機能**：全／未完了／完了 の切り替え |
| Bさん | **検索機能**：テキスト入力でTODOをフィルタ |
| Cさん | **件数表示**：上部に「全○件 / 完了○件」 |

※ 同じ `App.jsx` を触るので **コンフリクトの学習** にも適している

### 4. 各自実装＆PR

各自ブランチを切って実装：

```bash
git switch main
git pull origin main
git switch -c feature/自分の名前-機能名
```

実装（軽めで良い。ここでは流れの体験が目的）：

**Aさんの例（フィルタ）：**

```jsx
// App.jsx に追加
const [filter, setFilter] = useState('all')
const visibleTodos = todos.filter((t) => {
  if (filter === 'active') return !t.completed
  if (filter === 'done') return t.completed
  return true
})

// return内に追加
<div>
  <button onClick={() => setFilter('all')}>すべて</button>
  <button onClick={() => setFilter('active')}>未完了</button>
  <button onClick={() => setFilter('done')}>完了</button>
</div>
```

テストを追加：

```jsx
// App.test.jsx に追加
it('フィルタを「未完了」にすると完了済みは非表示', async () => {
  // ... テスト省略 ...
})
```

コミット＆push：

```bash
git add .
git commit -m "feat: フィルタ機能を追加"
git push -u origin feature/tanaka-filter
gh pr create --title "フィルタ機能追加" --body "全/未完了/完了 の切り替え。テスト追加。"
```

### 5. 相互レビュー

自分のPR以外の2件を開いてレビューする。

**GitHubのPRレビュー画面：**

1. **Files changed** タブで差分を読む
2. 特定の行にマウスオーバー → **+** アイコンで行コメント
3. **Review changes** ボタンから：
   - **Comment**（意見のみ）
   - **Approve**（承認）
   - **Request changes**（修正要求）

**レビューコメントのマナー：**

- **「なぜ」** を書く（「ここを〜にしたら？」→「〜だと△△なのでこうしては？」）
- **質問形で** 書く（相手の意図を尊重）
- **小さな提案と重大な指摘を分ける**（`nits:` prefixで軽微な指摘を明示する慣習）

**例：**

```
【良い】
このfilter関数、毎回新しい配列を作っていますが、
useMemo で囲うと再レンダリング時にキャッシュが効きます。
そうしない理由があれば教えてください。

【悪い】
useMemo 使えば？
```

### 6. コンフリクトを意図的に起こす

3人が同時に `App.jsx` を編集しているので、**最初の1本をマージした後、残り2本で必ずコンフリクト** が起こる。

**起きた時の解決：**

```bash
git switch feature/自分のブランチ
git fetch origin
git rebase origin/main
```

`App.jsx` にコンフリクトマーカーが出る：

```jsx
<<<<<<< HEAD
  <FilterButtons />
=======
  <SearchInput />
>>>>>>> feature/自分
```

**両方残したい時は両方書く：**

```jsx
<FilterButtons />
<SearchInput />
```

マーカーを削除して保存。

```bash
git add src/App.jsx
git rebase --continue
git push --force-with-lease
```

> **`--force-with-lease`** は `--force` の安全版。他人のpushを上書きしないよう最新チェックしてくれる。

### 7. 最終マージと動作確認

全員のPRがマージできたら：

```bash
git switch main
git pull origin main
npm run dev
```

**3機能すべてが動作する** かをチームで確認。Vercelプレビュー／GitHub Pages 本番URLでも確認。

**Actionsタブ** でこの日の実行回数を見る → 20〜30本のワークフローが走っているはず。

##  課題

### 基礎課題（必須）

1. 今日の相互レビューで **もらった指摘の中で一番役に立ったもの** を1つメモする
2. チームメンバーが書いたコードで「良いな」と思った書き方を1つメモ → 次回以降自分でも使ってみる
3. 次回までに「個人制作で何を作りたいか」を **3案** 考えてくる（発表アプリのアイデア）

### 応用課題（推奨）

4. **PR テンプレート** `.github/PULL_REQUEST_TEMPLATE.md` をチームで合意して作成する：

```markdown
## 変更内容
-

## 動作確認
- [ ]
- [ ]

## レビュー観点
-
```

→ 以降、PR 作成時に自動でこのテンプレが入る。**書くべきことを毎回書かせる仕組み**。

5. **Issue と PR を紐付け** してみる：
   - Issueを作成（例：「完了日時表示機能」）
   - そのIssueを close する PR の本文に `Closes #3` と書く
   - PRマージ時に Issue が自動でCloseされることを確認

→ タスク管理とコード変更の紐付けは、後からログを追う時の強力な武器。

6. **チームで「PRマージ条件」を正式にドキュメント化** する：承認数、CI必須項目、コンフリクト解消の責任者など。`docs/contributing.md` などに書く

> **狙い：** 暗黙のルールは忘れられる。**明文化** することで新メンバーでも迷わない運用になる。

### チャレンジ課題（挑戦）

7. **GitHub Projects（カンバンボード）** を1つ作り、各メンバーのタスクを Todo / In Progress / Done の3列で管理してみる。PRと自動連携する機能も試す

8. **「コードレビューで指摘しない方が良いこと」** をチームで議論する：
   - ただの好み（変数名の好み、空白の位置）→ Prettier / Lint に任せる
   - 原則論の長話 → ペアコーディングやミーティングへ
   - 別PRですべき大きな提案 → 別 Issue に切り出す

**PRレビューは「このPRをマージすべきか」の場** という原則を合意する。

9. 個人制作の **3案を相互に発表し合う**：チームメンバーから「どの案が一番面白そうか／実現可能か」のフィードバックをもらって、次コマの計画に活かす
