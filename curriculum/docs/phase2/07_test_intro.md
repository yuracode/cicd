##  本題

### 1. なぜテストを書くのか

**理由1：手動テストは回数が増えるほど割に合わなくなる**

- 機能が増える → 確認項目が増える
- 修正のたびに全部を手で確認できなくなる
- 「たぶん大丈夫」でリリース → バグ流出

**理由2：リファクタリング（コードを書き直す）の安全ネットになる**

テストが無いと、コードを整理した瞬間に「動いていたものを壊したかも」と怖くて触れなくなる。
テストがあると「テストが通れば動作は変わっていない」と自信を持って書き直せる。

**理由3：仕様の証拠になる**

テストコードは「このアプリはこう動くはず」の**実行可能な仕様書**。Readme以上に信頼できる。

**理由4：CIで自動チェックできる**（Phase 3 で扱う）

PRを出すたびにGitHubが自動でテストを回し、失敗したら警告を出してくれる。レビュアーの負担が激減する。

### 2. テストのデメリットも正直に

- **書くのに時間がかかる**（特に最初）
- **テストコード自体のメンテが必要**
- **過剰にテストすると逆に変更がつらくなる**
- **UIの見た目の崩れは検知しづらい**（見た目は人間の目が強い）

→ **「全部テストすればいい」は間違い**。重要な箇所・壊れたら痛い箇所から書く。

### 3. テストの種類

開発現場で使われるテストは大きく3段階。

**① 単体テスト（Unit Test）**

- 小さな関数やコンポーネント単体の振る舞いをチェック
- 速い・安い・たくさん書ける
- 例：「`addTodo('買い物')` を呼ぶと配列に1件追加される」

**② 結合テスト（Integration Test）**

- 複数の部品を組み合わせた時の挙動をチェック
- 例：「フォームに入力→送信ボタン→リストに表示 が流れる」

**③ E2Eテスト（End-to-End Test）**

- ブラウザを実際に自動操作して本物に近い動作確認
- 遅い・不安定・メンテ大変だが一番信頼できる
- 例：Playwright, Cypress

**テストピラミッド：**

```
         /\
        /E2\       ← 少なく（数件）
       /----\
      / 結合  \    ← ほどほど（数十件）
     /--------\
    /  単体    \   ← たくさん（数百件）
   /------------\
```

> **この授業で扱う範囲**：① 単体 と ② 結合 の中間。**Vitest** というテストランナーと **React Testing Library** を使う。E2Eは時間の関係でカバーしない。

### 4. テストを書く流れをイメージする

TODOアプリの `addTodo` をテストするイメージ（次回以降に実際に書く）：

```jsx
// こんな感じで書けるようになる（疑似コード）
test('addTodo で配列に追加される', () => {
  const todos = []
  const newTodos = addTodo(todos, '牛乳を買う')
  expect(newTodos.length).toBe(1)
  expect(newTodos[0].text).toBe('牛乳を買う')
})
```

**テストの3ステップ（AAA）：**

1. **Arrange（準備）**：入力データや状態を用意
2. **Act（実行）**：テスト対象を呼び出す
3. **Assert（検証）**：結果が期待どおりか確認

### 5. 手を動かす：`expect` だけを触ってみる

Viteプロジェクトの `package.json` がある場所で：

```bash
cd ~/workspace/todo-app
node
```

Node対話モードで以下を試す：

```javascript
// Node.js 標準の assert で試す
const assert = require('node:assert')
assert.strictEqual(1 + 1, 2)  // 通る＝何も表示されない
assert.strictEqual(1 + 1, 3)  // AssertionError が出る
```

> テストフレームワークは、これを **綺麗に書けて・まとめて実行できて・分かりやすく結果表示できる** ようにしたもの。中身の考え方は `assert` と同じ。

`.exit` で抜ける。

##  まとめ

### 今日分かったこと

- 自動テストは「手動確認をコード化したもの」
- 単体／結合／E2Eの3段階があり、この授業では前2つを扱う
- テストの基本は Arrange / Act / Assert の3ステップ

### よくある誤解

- **「テストは完成してから書くもの」** → 書けるなら機能実装と並行がベスト
- **「カバレッジ100%を目指すべき」** → 重要箇所の質が大事。数字を盲信しない
- **「テストが通れば品質保証」** → テストが見落としている範囲は当然ある。補助輪として使う

### 次コマ予告

次回から実際にコードを書く。Vitestをインストールして、簡単な関数のテストを書いて走らせる。

##  課題

### 基礎課題（必須）

1. 前回までのTODOアプリで「**ここをテストで自動化しておきたい**」と思う挙動を**3つ** リストアップしてくる（紙・メモアプリ何でも可）
2. 身近なアプリ（例：電卓、メモアプリ）を1つ選び、「もし自動テストを書くなら何を確認する？」を3つ書き出す

### 応用課題（推奨）

3. 1で挙げた3項目を、**AAA（Arrange / Act / Assert）パターン** で整理する。コードは書かなくていいので、各ケースを3行にまとめる：

```
Arrange: 前提条件（初期データなど）
Act:     操作（関数呼び出し、ボタンクリックなど）
Assert:  期待する結果
```

4. 2で挙げた3項目を **単体テスト／結合テスト／E2E** のどれに該当するかラベル付けする。迷ったら「何を切り分けて検証したいか」から逆算する

> **狙い：** テストを「なんとなく書く」のではなく、**ピラミッドのどこにいるか** を意識して書く習慣をつける。

### チャレンジ課題（挑戦）

5. Node.js に標準搭載されている **`node:test`** モジュールを調べ、`1 + 1 === 2` のアサーションを **Vitest を使わずに** 1本書いて実行する：

```bash
# ヒント：.mjs ファイルに書いて `node --test` で実行
node --test sample.test.mjs
```

→ テストフレームワークなしでも assertion は書ける、という感覚を掴むための体験

6. 「**テストがあっても防げないバグ** とは何か」を身の回りから1例考えてメモする（例：UIの見た目が微妙に崩れる、本番データとテストデータの違い 等）。テストの限界を知ることで、何を自動化すべきかが見えてくる
